计算机网络-自顶向下

CS基础
Publish Date:   2022-08-08

计算机网络

概述

相关术语：

  • 主机/端系统
  • ISP：网络提供商
  • RFC：标准文档，定义了一众协议
  • 协议： 定义了通讯实体之间交换报文的格式和顺序，语法、语义、时序、动作
  • 因特网：分布式应用以及为其提供服务的基础设施

因特网

  • 网络边缘：位于网络边缘的端系统以及应用程序
  • 网络核心：端系统分组交换机和链路构成的网络
  • 接入网：将端系统物理连接到边缘路由器的网络

交换方式：

  • 电路交换
    • 对所需的资源进行预留，分配好则他人不可使用，独占性
    • 对于同一条电路划分成不同的piece进行分配带宽
    • 复用方式：频分复用、时分复用
      • 频分复用：分配固定的频率（类比电台）
      • 时分复用：分配固定长度的帧和间隙
  • 分组交换
    • 端系统之间相互交换的信息称为报文，报文划分成小的数据块则称为分组(包，packet)
    • 越过通信链路使用全部的带宽
    • 复用方式：按需划分成时间片，非固定长度
    • 存储转发传输：在不同的节点之间，需要先将分组完整的存储起来，然后再转发到下一个检点，在此期间产生了延迟。总延迟为存储过程中产生的延迟和物理传输过程中的延迟(距离/速度)
    • 转发：通过目标来查询路由器，获取下一次转发的路径

延时：处理延时，排队延时，传输延时(上载下载速度)，传播延时(物理传播)，

协议层次以及模型

通过协议分层，只需关注当前层如何功能，对于下面的层次，只需调用其API即可，无需关注实现方式，(类似 controller service dao三层架构)

协议栈：

  • 应用层：网络应用程序以及相关协议存在的位置
  • 传输层：通过传输协议(TCP/UDP)进行应用层报文的传输
  • 网络层：IP协议传输数据报(datagram)
  • 链路层：将分组再不同的节点之间传输(MAC)，链路层分组为帧
  • 物理层：一个个bit的将数据进行传输，电信号，光信号表示01

OSI七层模型

多出了表示层和会话层

  • 表示层：使通信的应用程序能够解释交换数据的含义
  • 会话层：提供数据交换的定界和同步功能

封装

在将分组向下一层传输时，从上一次获取到有效负载(payload)，从在负载之前加上自己的首部字段

应用层

概述

应用层所运行的为应用程序以及其所支持的协议，如web项目，邮箱服务等以及所涉及的相关协议均运行在应用层之上

体系结构

  • 客户端-服务器（C-S）
  • P2P

进程通信

一个进程可认为是端系统上的一个程序，此处所说的进程通信为两个端系统之间的通信，通过交换报文的方式借助socket API，将报文下交给传输层，使用TCP/UDP进行传输

定位方式

一个进程对应着一个端口，通过主机IP+端口即可，具体绑定方式在传输层中讨论，对于应用层只需调用socket API即可

应用层协议

HTTP

  • 定义了浏览器向服务器的请求方式，以及服务器做出相应，传递页面的方式

  • 以tcp作为支撑

  • 无状态协议

  • 默认为持续连接，即创建tcp连接(三次握手)后保持该连接

三次握手

  1. 客户端向服务器发送tcp报文段，
  2. 服务器用tcp报文段确认并作出响应
  3. 客户端向服务器返回确认

通过该三次握手，第二次握手服务端确认了客户端的发送能力，客户端接收到服务端的响应后确认了服务端的接收和发送能力，第三次握手服务器确认了客户端的接收和响应能力，从而客户端和服务端的发送和接收能力都得到确认，建立起完备的连接

HTTP报文

  • 请求报文：

    • 请求行：方法字段 url字段 http版本字段

    • 首部行：host，user-agent等，F12抓包查看即可

  • 响应报文

    • 初始状态行
    • 首部行
    • 实体

C-S交互

  • cookie&session
  • JWT

Web缓存

正向代理服务器，在客户端侧，客户端和代理服务器建立tcp连接后向代理服务器发送http请求，若代理服务器本地有，则直接返回给客户端即可，起到了缓存的效果，如果没有，再向目标服务器建立连接发送请求。

  • 代理服务器位于局域网内，客户端与代理服务器之间比和原目标服务器速度要快
  • 源目标服务器被访问次数减少，压力减小，响应速度变快

HTTP 1.1

HTTP1.1主要在长连接方面进行了优化，使得GET POST请求后连接不断开，无需再次重新建立连接,但是同时又引入了队头阻塞的问题，即某个HTTP请求的响应没有被即时的返回，后面的响应都会被阻塞

除此之外还有部分传输、HOST、缓存等方面

HTTPS

通过SSL进行加密，HTTPS解密出来仍是HTTP

非对称加密

HTTP2.0

并行进行请求，发送多个请求，提高访问速度。原本的HTTP1.0更像是个停等协议，2.0由停等转变为流水线方式

QUIC/HTTP3.0

  • 使用UDP
  • 在用户态Kernel
  • 0RTT

电子邮件协议

电子邮件系统主要由三个部分组成：

  • 用户代理
  • 邮件服务器
  • 简单邮件传输(STMP)

电子邮件协议主要分为发送协议和访问协议：

  • 发送协议：STMP
  • 访问协议：常用 POP3和IMAP

STMP

发送过程：

  1. Alice调用用户代理程序撰写报文，并提供Bob的邮箱，指示用户代理发送该报文
  2. Alice用户代理把报文发送给邮件服务器，报文放在报文队列当中
  3. Alice的邮件服务器上的STMP客户端发现报文，通过TCP与Bob的STMP建立连接，发送报文
  4. Bob的邮件服务器上，STMP服务器接收报文，存入邮箱

邮件服务器为一个实体服务器，而STMP服务器邮件服务器上运行的应用程序，类似于http服务器

POP3

分为三个阶段：

  1. 特许(Authorization)：对用户进行鉴权
  2. 事务处理：用户代理代理取回报文 （删除）
  3. 更新：客户端发出quit命令，结束会话

最新的POP3协议可以通过配置来选择“下载删除”或”下载保留”

IMAP

将报文和文件夹相关联

IMAP协议中客户端和服务器上的邮件保持同步，如果不⼿动删除邮件，那么服务器上的邮件也不会被删除。IMAP这种做法可以让⽤户随时随地去访问服务器上的邮件。

HTTP

基于Web的电子邮件，用户代理即为Web浏览器，用户和远程邮件服务器之间的通信则使用HTTP协议，而非POP3和IMAP，邮件服务器之间的仍为STMP

DNS

作用：IP地址与域名之间的解析

组成：

  • 分层的DNS服务器实现的分布式数据库
  • 使主机能够查询分布式数据库的应用层协议

分布式，分层数据库

DNS服务器主要分为三个层级：

  • 根DNS服务器
  • 顶级域服务器(com,org,io,edu等)
  • 权威服务器：组织机构保存对外公共访问的域名和IP的解析关系

查询方式：

  • 递归查询：首先向根目录发起查询请求，根目录在向顶级域发起查询，最终到提供的域名的权威服务器，权威服务器再将结果向上传达，最终由根将结果告知给用户
  • 迭代查询：首先向根目录发起查询请求，根目录让用户向顶级域去发起请求，顶级域再让用户向权威服务器发起查询，最终由权威服务器将查询结果告知给用户

二者之间的区别为最终由谁将结果告知给查询的用户，递归为向下递归查询，再将查询的结果向上传递，由最初的根将结果返还给用户，而迭代更像是再对其进行重定向

DNS记录和报文

数据库中存储的资源记录格式如下：(Name,Value,Type,TTL)

其中，Name和Value的具体值由Type来决定：

  • Type = A Name为主机名 Value为IP
  • Type = NS Name为域名 Value为DNS服务器的主机名
  • Type = CNAME Name为主机的别名，Value为规范主机名
  • Type = MX Name为别名，Value为规范主机名（邮件服务器）

  • 提供主机别名（host aliasing）服务，有着复杂主机名的主机可以有一个或者多个别名，例如aaa.xxx.com的主机可能还会有aaa.com和www.aaa.com两个别名，在这种情况下，aaa.xxx.com叫做**规范主机名**（canonical hostname）。主机别名的特征是比规范主机名更容易记忆，DNS可以提供根据主机别名获取规范主机名的服务

  • 提供负载分配服务（load distribution） 一般来说，被繁忙访问的大型站点是分布在多台服务器上的， 这个时候，主机名和IP地址就不是一一对应的关系，而是一个主机名对应一个IP地址的集合。 在大量的，连续的多次访问中，DNS通过旋转IP地址达到负载均衡的目的：在向这个主机名发出DNS请求的时候，服务器会用包含全部这些IP地址的报文进行回答， 但在每个不同的回答中会旋转这些IP地址的摆放顺序，而客户机总向报文中排在最前的IP地址发出请求

当查询时，提供一个域名，通过域名，在数据库中查询Type = NS类型的资源记录，找到对应的DNS服务器，在向DNS服务器进行查询，在其中找到 Type = CNAME的记录 查询出主机名，在通过主机名查询Type = A的资源记录，最终查询出域名对应的IP

插入记录

向注册登记机构提供 ip和基本/辅助DNS服务器的名字，将Type = A，NS类型的插入到数据库当中们同事还要确保A类型和MX类型要存储在自己的权威DNS服务器当中

P2P

P2P模式避免了传统C-S模式下服务器性能瓶颈的情况，每个节点既可以请求，也可以提供服务

速度分析：

  • 最初只有服务器有文件，因此需要通过服务器发送给对等方${F}/{U_{s}}$F为文件大小，Us为服务器上载速度,
  • 在对等方下载：$F/d_{min}$ dmin为最低下载速度
  • 当有N个一同请求时：总的传输为$NF$，总的上载能力为服务器的上载能力加上N个对等方的上载能力：$u_s+∑_{i=1}^{N}u_i$

因此，最小分发时间为$D_{P2P} = max[{F}/U_{s},F/d_{min},{NF/(u_s+∑_{i=1}^{N}u_i)}]$

BitTorrent

洪流：参与一个文件分发的所有对等方的集合

一个对等方参与到了洪流当中，下载的同时也在进行上载，在他完成下载后，可以选择离开洪流，也可以选择继续在其中无私奉献

Alice周期性的询问邻居的具有的块列表

策略：

  • 请求：最稀缺优先，Alice首先请求在邻居中副本最少的块
  • 响应：优先响应以最高速率给Alice提供服务的邻居，同时每过30s随机选出一个邻居来提供服务

CDN

内容分发网，通常用于进行流媒体访问加速

安置原则：

  • 深入：靠近用户
  • 邀请做客：在少量关键位置设置大集群来邀请到ISP做客

CDN通过拦截客户端的请求来确定适合该客户的CDN服务器集群，同时将请求重定向到该集群。

运输层

要点：

  • 多路复用/解复用
  • 可靠数据传输
  • 流量控制
  • 拥塞控制
  • TCP/UDP

概述

运输层协议为不同主机上的应用进程之间提供了逻辑通信功能，在应用层的角度，直接通过socket编程的方式即可完成不同应用程序之间的通信，而不需要关注底层的网络层和链路层物理层等

运输层从应用层出接收到报文，划分成不同的段并添加上头部，形成运输层报文段，向下传递后发送给对方，对方再把报文段还原成报文上交给应用层，应用层收到报文后即为一次通信结束

多路复用解复用

应用程序-进程-socket-端口之间一一对应(不算进程fork出来的子进程)

网络层提供的为端到端的服务，运输层则为进程到进程的服务，因此，为了通过网络层传输运输层的报文段，实现进程到主机的对应关系，便有了多路复用/解复用

多路复用：

  • 一个socket套接字对应着一个进程，为了让不同的进程的数据均通过网络层的IP协议传输，
  • 因此在源主机从不同的套接字中手机数据块，并为每个块封装上首部信息，生成报文段，将报文段传递到网络层即为多路复用
  • 生成报文段之后，在网络层看来，不断进程的数据在性质上已经相同

多路分解：将运输层报文段中的数据交付到正确套接字出，重新在进程层面上加以区分

经典案例：Ann和Bill两家之间写信通信

多路复用要求：

  • 套接字有唯一标识符
  • 每个报文段有特殊字段(源端口+目标端口)来指示该报文段要交付到的套接字

TCP使用的为四元组（源IP，源端口，目标IP，目标端口）作为特殊字段来绑定和表示套接字，完成数据到套接字的正确指派

UDP只有二元组(目标IP,目标端口)

UDP

UDP的主要特点为无连接，并且出了多路复用和解复用和少量误差检测之外，几乎没有在IP之上添加新的功能，没有像TCP的可靠性校验和阻塞控制等，因此保证了低延时等特点，被运用于流媒体，DNS等对延时敏感的场景上

  • 发送什么数据和什么时候发送的应用层更为精细

  • 无须建立连接

  • 无连接状态

  • 分组首部开销小

UDP报文段结构

其中，校验和为源端口号+目标端口号+长度的反码，因此源端口号+目标端口号+长度+校验和的结果各个位全为1，全为1则通过校验

提供检验但是对擦错无恢复能力

可靠数据传输原理(RDT)

停等协议

RDT1.0

假设底层通信完全可靠，因此在运输层不做任何可靠保障，但IP提供的为尽力而为的服务，并不提供任何可靠性保障，只是一个美好的念想

RDT2.0

  • 发送方等待接收方的ACK或NAK
  • 停等协议
  • 无超时重传且无法处理ACK和NAK受损的情况

RDT3.0

  • 加入超时重传 发送方发送后启动定时器 在分组和ACK丢失时会超时重传
  • 依然为停等协议
  • ACK通过 0 1表示受到了的分组的序号

流水线协议

为解决停等协议的性能和信道利用率问题，提出了流水线协议，一次发送多个报文段，根据异常处理的方式不同又可分为GBN和SR

  • 发送缓冲区 >1 接收缓冲区 = 1：GBN
  • 发送缓冲区 >1 接收缓冲区 > 1：SR

而发送缓冲区和接收缓冲区均为1时即为停等协议

Go Back N：

发送方可发送多个分组，但是接收方只能确认一个，如果已经接收到了第N个，但是第一个丢失，那么从第一个开始重发

可将分组分为四组：

  • 发送且已经确认
  • 发送了但还未确认
  • 还未发送但是处于发送的缓冲区内
  • 还未进入发送缓冲区，不可用

Selective Repeat

相比GBN多了接收缓冲区，在接收缓冲区内任一一个分组丢失，只需要对该分组进行重发

TCP

连接管理

  • 逻辑连接
  • 全双工服务
  • 报文段的数据量受限于最大报文段长度(MSS),最大传输单元(MTU)[最大链路层帧长度]

三次握手：

  1. 客户端现象服务端发送一个TCP报文段，不包含应用层数据，SYN位置为1
  2. 服务端提取SYN报文段，为该TCP连接分配资源，像客户端发送允许连接的报文段
  3. 客户端分配完资源后，服务端发送一个确认连接的报文段，至此三次握手结束

第一次握手服务端确认了客户端具有发送报文的能力，第二次握手客户端确认了服务端的接收和发送能力，第三次握手服务端确认了客户端的接收能力，双方均确认对方的发送和接收能力 –来自《码出高效》

四次挥手：

双向关闭：

  • 一二次挥手：客户端向服务端发送FIN并接收到ACK后，客户端到服务端的通道已经关闭，但此时服务端还可向客户端发送
  • 三四次挥手：服务端向客户端发送FIN并接收到ACK后，服务端到客户端的通道也进行关闭，服务端接收到这个请求后，服务端就能成功变成关闭（CLOSE）状态。客户端则会等一段时间再进入关闭状态，因为第四次挥手不一定能成功发给服务端，所以要等一下，看看服务端会不会因为没收到第四次挥手，而重发第三次挥手。

报文段结构

  • 序号：该报文段的首字节的字节流编号(应用层向运输层提供的为字节流)
  • 确认号：期望收到的下一字节的序号
  • 接收窗口字段：用于流量控制
  • 可选字段：用于发送方和接受方协商MSS

RTT

  • 往返时间，可用于在超时重传时作为超时时间的参考

  • 动态计算：

    $EstimatedRTT = (1-a)* EstimatedRTT +a*SampleRTT$

    对于当前测量的RTT给予较小的权重，通常设置a=0.125

  • DevRTT：用于估量SimpleRTT相对EstimatedRTT的偏移程度

    $DevRTT = (1-β)DevRTT+β|SampleRTT-EstematedRTT|$

  • 超时时间通常设置为：$TimeoutInterval = EstimatedRTT+4 * DevRTT$

EstimatedRTT和DevRTT类似于正态分布中的u和σ的关系，$EstimatedRTT+4 * DevRTT$意为在概率为XX的情况下，该分组已经丢失

可靠数据传输

大致与之前讨论的流水线方案类似，但有针对调整设计

  • 超时间隔加倍：TCP每次重传后都将超时时间设为原来的两倍，而不是通过RTT估算的来的

  • 快速重传：收到三个冗余ACK，则快速重传，不再等待超时定时器超时

  • 丢失控制：

    • 发送方仅维持发送了但未确认和下一个要发送的

    • 接收方发送ACK时采用一个辅助定时器，在定时器到时前不发ACK，将时间段内多个ACK一同发送给发送方

    • 接收方仍设置缓存：在中间存在丢失分组，形成Gap时，返回ACK X，要求接收方来填补Gap

流量控制

接收方维护两个变量：

  • LastByteRead：从缓冲区中读出的最后一个字节编号
  • LastByteRcvd：缓冲区中的最后一个字节编号

二者相减即可得到缓冲区中剩余的字节数，要求剩余的字节数小于缓冲区大小

$LastByteRcvd-LastByteRead <= RcvBuffer$

接收窗口rwnd:还能继续接收的字节数

$rwnd = RcvBuffer -[LastByteRcvd-LastByteRead]$

接收窗口这个变量交给发送方来维护，从而控制进行流量控制

同时发送方维护：

  • LastByteSent：发送的最后一个
  • LastByteAcked：确认的收到的最后一个

二者相减即可得到发送出但还未收到的字节数，要求这个字节数小于接收窗口的大小，即可保证接收方能正确处理

$LastByteSent-LastByteAcked <= rwnd$

拥塞控制

发送方维护者一个拥塞窗口(cwnd)的变量，对TCP发送方的速率进行限制，有：

$LastByteSent-LastByteAcked <= min(rwnd,cwnd)$

若接收方缓存足够大,则有：

$LastByteSent-LastByteAcked <= cwnd$

TCP将丢包事件(超时或三个冗余ACK)视为阻塞的证明

TCP拥塞控制算法：

参数：

  • cwnd
  • ssthresh：慢启动阈值，用于结束慢启动模式
  1. 慢启动：cwnd最初设置为一个MSS的大小，只要发送被确认就×2，以指数增长，检测到拥塞时将ssthresh设置为cwnd的一半，cwnd再重新从1开始继续慢启动，只要在cwnd和ssthresh相等时，慢启动模式才结束，转入拥塞避免阶段
  2. 拥塞避免：每个RTT将cwnd+1,出现拥塞时同样将ssthresh设置为cwnd的一半，而对于由三个冗余ACK引发的拥塞，转入快速恢复阶段
  3. 快速恢复：收到一个冗余ACK，cwnd增加一个MSS

最终呈现的结果为线性增(cwnd每次增加一个MSS)，乘性减(ssthresh变为cwnd的一半):

公平性：

对于两个TCP连接公用一条链路的情况，如果假设跳过了慢启动阶段，则拥塞窗口按照线性增，乘性减的方式，到达阈值之后二者一同÷2后+1，即便最开始二者的吞吐量相差悬殊，多次÷2+1后最终区域相同，达到公平的平衡状态

A为起始状态，A->B->C->D,最终趋于相同

而对于UDP，UDP不存在拥塞控制，因此可能出现UDP压制TCP流量的情况

网络层：数据平面

网络层概述

网络层主要分为数据平面和控制平面两个平面，分别对应着转发和路由选择的功能

控制平面通过路由选择算法为每台路由器计算出转发表(传统或SDN的方式)，路由器再根据网络层分组的首部字段，作为索引在路由表当中查询出对应的value，即为输出链路接口，即可交给下一个路由器

路由器

主分为四个部分：

  • 输入端口：查找转发表确定输出端口，同时还执行一定的物理层和链路层的功能
  • 交换结构：将输入结构和输出结构相连，一个”网络结构”
  • 输出端口：存储从交换结构而来的分组，并执行物理层和链路层的功能进行传输
  • 路由选择处理器：传统路由器中负责执行路由选择协议，SDN控制方式中负责与远程通信，接收传来的流表

在输入端口出进行查找，确定了输出端口之后再进入交换结构，向输出端口传输

此处的端口并非在应用层所说的软件端口，而是路由器的物理接入和输出接口

输入端口处理

在输入端口出执行路由表的查询，来寻找输出端口，转发决策在本地做出，无需调用集中式路由选择处理器，避免了集中式处理的瓶颈

通过最长前缀匹配原则，将一定范围内的IP地址和路由器的输出接口做映射，为保证速度，采用硬件逻辑

其他工作：

  • 进行物理层和链路层的处理
  • 检查分组的版本号，TTL，校验
  • 更新用于网络管理的计时器

交换

  • 经内存交换：在CPU的控制下直接完成
  • 经总线交换
  • 经互联网络交换：通过控制线路的导通和闭合来确定线路

输出端口处理

分组调度

当传输速度超过处理速度时，便会出现排队与阻塞，即分组位与输出和输入接口的缓存当中，而当缓冲区满时，则会出现丢包，网络层为尽力而为的服务，不会对丢包进行任何处理，可靠性交给运输层的TCP来进行保障

此时便需要相应的调度算法来确认分组的处理顺序

  • FIFO
  • 优先权排队：缓冲区分为多个队列，队列内FIFO，队列之间优先处理高优先级的
  • 循环排队：分为多个类别的队列，队列内FIFO，队列之间循环选择
  • 加权公平排队：分为多个权重不同的队列，分组按照权重分配给不同的队列，导致各队列内分组数量不同，队列之间循环选择。

路由选择算法

本质上就是解决图的最短路径问题，可通过集中式或分布式来划分：

  • 集中式路由选择算法：在真正开始计算前，需要对全局的cost已知，也叫做链路状态算法，代表为Dijkstra算法
  • 分布式路由选择算法：计算开始时各个路由器只知道自己邻居的状态，没人清楚全局的状态。路由器通过迭代的方式，不断告诉邻居相应的状态，从而最终获取全局的状态，确认路径

链路状态路由选择算法-Dijkstra算法

本质为迭代算法，经过N次迭代可以知道从起始节点到N个节点的最短开销路径，核心思想为贪心算法，通过局部最优解来推算出全局最优解

例：

求以u为起点，到各个点的最小开销路径

具体步骤：

  1. 初始化：与u相连的个点的距离设置为图上标注的cost，与u无法直达的点初始化为∞，u放置在已知点的队列之中，其他点放置在未知点队列之中
  2. 第一次迭代：选取一个距离最短的点，加入到已知节点当中，在第一次选择的便是·x,cost为1，此时的已知队列变为ux
  3. 第二次迭代：由于x已经为已知节点，便可计算通过x出发的消耗，再加上已知的ux之间距离，即可计算出该节点到u的距离，例w通过x到达比u直达更短，对其进行更新，y通过x变为可达节点，也对其进行更新
  4. 重复迭代直至所有的节点均变为已知节点

距离向量路由选择算法-Bellman-Ford方程

迭代、异步、分布式的算法：每个节点都从邻居那获取信息，执行计算，然后将计算结果发送给邻居，一直持续到再无更多信息要交换为止

Bellman-Ford方程

$d_x(y) = min_v[c(x,v)+d_v(y)]$

$d_x(y)$:从节点x到节点y的最小开销

c(x,v) :节点v到x的开销

$d_v(y)$:节点v到节点y的最小开销

$min_v$所针对的为x的所有邻居

具体步骤：

  1. 最初xyz仅知道到自己邻居的直接距离
  2. y将自己的信息传递给x之后，x计算以y为中间节点到达z的距离，并进行更新，y z同理进行计算更新
  3. 再将新计算出信息继续进行传递，(x通过y的信息计算出新的信息传递给y，z)
  4. 直至最后没有新的信息产生

好消息传得快，坏消息传得慢

  • 对于cost由4变为1的好消息：

    1. y检测到链路发生变化，对其进行更新
    2. y通知z y至x的距离进行了缩减，z进行重新计算
    3. z将重新计算后的结果发给y，y无需进行更新，至此算法进入静止状态，结束迭代

  • 对于cost由4变为60的坏消息：

    1. y计算到x的开销：min(60,6) = 6
    2. y将消息传递给z，z无需进行更新，进入静止状态

    但如果x z之间的开销为50：

    1. y计算到x的开销：min(60,6) = 6
    2. y将消息传递给z，z到x的开销为min(50,1+)

==存疑==

LS DV二者之间比较：

  • DV仅与他的邻居进行交谈，LS需要全局信息
  • 收敛速度：LS要求O(NE)个报文复杂度为O(N^2) 的算法，DV需通过迭代，收敛速度较慢，还会遇到无穷计数的问题
  • 健壮性：DV算法情况下，如果一个节点发生问题，则会导致迭代异常，从而影响全局，不正确的结果会扩散至整个网络，LS使用的Dijkstra算法，处理的为单源问题，故计算的只是自己路由器到其他节点的开销，每个路由器之间为独立计算，健壮性较好

路由协议

在进行路由时，基本以子网为单位，全网几百万的子网来执行距离矢量算法或者链路状态算法开销依旧是承担不起的，因此，提出了自治区的概念：

  • 在自治区内部按照自治区内部的路由选择算法来进行路由，使用RIP、OSPF或者私有的算法
  • 在自治区之间，通过BGP算法，将各个自治区(ISP)粘合起来

这样，即便是新加入了一个一个新的子网，配合上路由聚合，子网内的多个IP对于全网来说也就是多了一个点，大大减轻了经路由选择算法的负担

自治系统内部的路由选择

RIP

  • 基于距离矢量算法
  • 代价为跳数
  • 跳数最大为15跳，16跳视为不可达
  • 最多25个目标子网，适用于小网
  • 每隔30秒交换报文，或者请求时交换报文

OSPF

  • 使用泛洪链路状态信息和Dijkstra算法
  • 链路的代价由管理员配置
  • 链路状态发生变化时，广播链路状态信息，未变化则每隔30min进行一次广播
  • 优点：
    • 安全
    • 当存在多条开销相同的路径，允许使用多条路径，减小单条路径的压力
    • 可层次化配置多个区域，每个区域运行自己的链路选择算法

ISP之间路由选择：BGP

RIP和OSPF完成了自治系统内部的，BGP进行自治系统之间的路由，将多个ISP黏合起来

可以理解成距离矢量算法的升级，通知的信息更加全面，在BGP协议当中，不仅向邻居通告开销，并且向邻居进行通告路径

例如此时3b上线，3a路由器向1c路由器通告 AS3 3b，1b路由器向2a路由器通告AS3 AS1 3b,此时 AS1 AS2自治系统均知道了AS3中的3b节点上线，并且可以通过AS1 AS3抵达

  • 跨越两个AS的连接称为外部BGP(external BGP eBGP)
  • AS之内的称为内部BGP(internal BGP iBGP)

例如3a路由器，既可以作为网关BGP之间进行通告，又可以作为内部路由器参与内部路由算法

  • NEXT-HOP:下一个AS的网关路由器的接口，在上图中，对于AS3向2b发送，AS1中的1c即为NEXT-HOP
  • AS-PATH:包含通告已经经过了的AS的列表，在上述例子中，AS2到子网3b，AS-PATH为 AS1 AS3

热土豆算法

字面意思，尽快将手上的热土豆尽快抛出，尽快将分组送出AS因此选择通往NEXT-HOP路由器的最小开销的路径

热土豆算法只考虑了在自己的AS内的最小代价，而并未考虑AS之间传输的代价，因此，最终取得的路由选择结果并非最优结果，只是一条开销较小的路径

实际应用

  1. 路由器被指派了一个本地偏好作为其属性之一，该属性完全由管理员负责，具有最高本地偏好值的路由将被选择
  2. 当本地偏好相同时，选择具有最短AS-PATH的路由
  3. 本地偏好和AS-PATH均相同时，使用热土豆算法

SDN控制平面

体系结构特点：

  • 基于流的转发
  • 数据平面和控制平面相分离，其中控制平面由一个控制器和若干个网络控制应用程序组成
  • 网络控制功能
  • 可编程网络 ：网络控制应用程序为可编程1的

相比于传统的网络层 SDN最大的特点即为分层，原本交换机/路由器为一个整体，只能由一家厂商生产，而对于SDN的方式，不同层次的服务可以由不同厂商来提供，在生态系统上更加开放

控制器组成：

  • 通信层：用户在控制器和SDN控制器控制的交换机、主机等之间传送信息，之间通过南向接口
  • 网络范围状态管理层：SDN控制平面所做出的最终控制决定，SDN控制设备的最新状态信息等
  • 对于网络控制应用程序层的接口：控制器和网络控制应用程序之间通过北向接口进行交互(REST风格)

SDN控制器在逻辑上集中，但是具体实现上为分布式的，如服务或数据库采用分布式结构

循环冗余检测

规则：

  • 将数据为左移r位，并添加r比特的附加比特，形成最终要传输的值

  • 双方约定一个生成多项式，称为G，G的最高位必须为1

  • 在接收方，如果收到的值进行模2运算(异或)恰好能被G整除，则大概率不存在错误

因此，现在的问题变成了如何求出R来保证合成的数据能被G整除

公式推导：

如果能被G整除(模2运算)，那么则有：$D*2^{r} ⊕ R = nG$（D左移后空出的位均为0，0与R异或结果为R，相当于把R补在了后面）

对公式进行变化，两边同时异或R：$D*2^{r} ⊕ R ⊕ R = nG ⊕ R$

又因为：$R ⊕ R = 0$，则有：$ D * 2^{r} ⊕ 0 = nG ⊕ R$

$D * 2^{r} = nG ⊕ R$,该式子表明：$D * 2^{r}$除nG得到的余数为R

$R = remainder D * (2^{r}/G)$

多路访问链路和协议

在广域网之上，链路层采用的是点到点的传输，像太平洋之下的海底电缆，纽约直通上海，二者之间直发

但是对于局域网来说，采用两两点对点连接的形成的网状过于复杂，因此在局域网之上采用的方案为多个设备连接到同一个共享设备之上，一个设备进行发送，采用的为广播的方式，所有的其他节点都能收到，但这也引入了两个问题：

  • 多个节点同时发送时会产生碰撞，好比多个人同时说话而互相听不清
  • 如何辨别广播中的信号是发送给自己的

辨别广播信号是否是发送给自己的，则引入了MAC地址，通过比较帧中的目标MAC，来判断自己是否应该接收

为解决碰撞问题，引路了多种多路访问协议，主要有：

  • 信道划分协议
  • 随机接入协议
  • 轮流协议

信道划分协议

  • 时分 TDM，每个节点只有在在允许自己发送的时隙才能发送
  • 频分 FDM，每个节点使用自己的频率，类似收音机
  • 码分 CDMA 每个节点使用自己的编码

时隙为真正的时间划分单位，每个节点发送一个时隙后切换为下一个节点

帧为统计周期，信道的传输速率为R，N个节点时，在一个帧内，一个节点的发送速率为$R/N$

无论是时分，频分，还是码分，存在的问题都是一个节点的发送速率被限制在了$R/N$,无法有效利用信道容量来全速发送，随机接入协议即可解决该问题

随机接入协议

一个节点总是按照全速来进行发送，当有碰撞时，涉及碰撞的节点反复重新发送，直至无碰撞的通过为止。但是如果多个节点都同时重新发送，那么碰撞的问题则会一直存在，形成阻塞，因此引入一个随机时延，每个节点在经过一个随机时延后再重新发送，将各个节点的发送错开

而对于随机接入协议，也可继续细分

时隙ALOHA

做出如下假设：

  • 所有帧由L比特组成
  • 时间被划分成L/R的时隙(一个时隙等于传输一帧的时间)
  • 节点在时隙开始时传输帧
  • 节点时同步的，每个均知道时隙什么时候开始
  • 一个时隙中有两个或者更多个帧碰撞，所有节点在时隙结束前能检测到

设定一个概率p，操作逻辑如下：

  • 当节点有新帧需要发送时，在下一个时隙开始时传输整个帧
  • 没有碰撞则成功传输
  • 如果存在碰撞，该节点在时隙结束之前检测到，在后续的时隙按照p进行重传

通过按照概率重传，将碰撞错开，避免阻塞，并且在传输时可以按照满速进行传输

性能分析：

现有N个节点，根据N重伯努利实验，一个节点成功发送的概率为$p(1-p)^{N-1}$,

N个节点中有一个能成功发送的概率为$N*p(1-p)^{N-1}$

最大效率为37%,即37%的时隙在做有用的工作

ALOHA

除无时隙之外其他和时隙ALOHA思想一致

由于无时隙，任意时间都可进行传输，如果不存在碰撞，则在$[t_0-1,t_0+1]$时间段内均不能有其他节点进行传输，对应的概率为$p(1-p)^{2(N-1)}$

最大效率为时隙ALOHA的一半

载波侦听多路访问

  • 先对信道进行侦听，如果检测一小段时间没有其他节点进行传输，该节点再进行传输
  • 检测到碰撞，自己则停止传输

即便事先已经进行了侦听，但是由于传输时延的问题，依旧会产生碰撞：

具有碰撞检测的载波侦听多路访问

在发生碰撞后，中止传输，等待一个随机量，再重新侦听链路，准备发送，之所以使用随机量，同样是为了将多个碰撞的节点相互错开，和使用概率p的目的一致

在用于以太网时，做出了细微调整，当该节点发生了n次碰撞，则从$[0,1,2,….2^{n-1}]$中选择一个作为等待时间，碰撞的次数越多，等待时间长的可能性就越大，从而更大程度的保证错开

轮流协议

轮询协议：

选择一个节点为主节点，主节点按照循环的方向轮询每个节点，主节点观察信道上是否缺乏信号，来决定一个节点何时能发送帧

缺陷：

  • 主节点寄则协议寄
  • 引入了轮询时延

令牌传递协议：

使用token在节点之间按照次序进行传递，只有持有令牌时才能有资格发送，发送后将token传递给下一个节点

交换局域网

寻址

MAC地址

拥有链路层地址实际上为网卡，而非主机或路由器

链路层的交换机不具有和接口相对应的链路层地址，(而对于路由器而言，每个接口均有对应的IP地址，转发即是寻找正确的接口),因此交换机透明的执行承载数据报的任务，主机和路游戏寻址也不会寻址到交换机处

并且，MAC地址解决了之前提及的如何在一个广播的链路中确定该帧是否是发给自己的问题，通过将帧中的目标和自己的MAC地址进行匹配，如果不匹配，则丢弃该帧

地址解析协议ARP

每台主机和路由器在内存中包含一个ARP表，ARP表包含IP和MAC地址之间的映射关系

通过ARP协议，以下一跳的IP作为输入，返回相应的MAC地址

在子网内，通过广播的方式，向各个主机和路由器发起查询，以求获取对应的MAC地址

跨域子网发送

以111.111.111.111向222.222.222.222发送为例：

  1. 首先111.111.111.111想向外发送，首先通过IP的路由选择算法确定了下一跳的IP为网关路由111.111.111.110
  2. 再通过ARP协议，查询到该路由器接口的MAC地址，此时便可以将帧进行发送
  3. 通过路由器的转发表确定通过接口222.222.222.220向外进行转发
  4. 该接口再通过ARP通过目标IP来查询对应的MAC地址：49-BD-D2-C7-56-2A
  5. 该接口对帧重新进行封装，源MAC为1A-23-F9-00-11-BB-4B,目标MAC为49-BD-D2-C7-56-2A
  6. 最终，222.222.222.222收到了该帧，解析出IP数据报向网络层进行传递

以太网

有线局域网

帧结构：

  • 数据字段承载IP数据报
  • 类型：网络层协议名
  • CRC：循环冗余校验

由于有线网本身较为可靠出错率极低，故以太网提供的为不可靠的服务，CRC通过时不确认，未通过也不否定，依靠网络层来提供可靠服务(TCP)

链路层交换机

接收入链路层帧并转发到出链路

交换机本身无MAC地址也不会被寻址，透明进行工作

转发和过滤

借助交换机表：

  • MAC地址
  • 接口
  • 表项放置的时间

如果在交换机表中找不到对应的匹配项，或者匹配到的为输入接口，进行过滤

自学习

交换机表为自动、动态。自治的建立起来的，无需管理员进行干预

  • 交换机表初始为空
  • 对于每个接口接收到的如真，交换机表将其存储在其中
  • 在一段时间后(老化期)，如果没有接收到以该地址作为源地址的帧，将其从中删除

交换器和路由器

  • 路由器使用网络层地址转发分组，交换机使用MAC地址转发分组
  • 交换机为第二层分组交换机，路由器为第三层
  • 交换机只是根据交换机表进行MAC地址的匹配和转发，自身并无地址，路由器的接口存在IP地址和MAC地址，参与路由选择算法

虚拟局域网

支持VLAN的交换机允许在一个物理局域网中定义多个虚拟局域网

对于交换机的端口进行分组，一个组分为一个虚拟局域网，彼此之间相互通信，不同组之间的帧彼此隔离

而不同的VLAN之间想要进行通信则需要借助一台外部路由器，逻辑上时一个LAN跨域了路由器抵达了另一个LAN，在硬件上，通过将交换机和路由器进行集成，合成单一设备，省去了外部路由器

数据中心网络

  • 主机分布在不同的机架上，每个机架上设置一台交换机，连接主机和其他交换机
  • 设置边界路由器，和外部因特网相连
  • 等级体系结构
  • 负载均衡：外部请求首先被定义到负载均衡器，负载均衡器再将请求分发到对应的主机上，类似于NAT或反向代理服务器

当前的等级体系结构在内部流量的转发存在一定的限制，如机架1 2 3 4 分别向 5 6 7 8发送请求，则A->B,B->C则成为了瓶颈，故向全连接的方向演变

Web请求历程

1.DHCP、UDP、IP和以太网

当主机和网络连接时，首先需要做的便是获取IP，因此采取的第一个动作便是运行DHCP协议，将自己接入到网络当中

  1. 生成一个DHCP请求报文 放入UDP报文段，该UDP报文段在放入到目标IP为255.255.255.255(广播),源IP为0.0.0.0的IP数据报
  2. 将该IP数据报封装到目标MAC为FF:FF:FF:FF:FF:FF的以太网帧中，进行广播，由于MAC地址和网卡绑定，故源MAC即为用户主机的MAC地址
  3. 由于目标MAC为广播，因此交换机向所有的出端口广播，最终达到路由器
  4. 路由器从以太网帧中抽出IP数据报，目标IP为255.255.255.255指示了这个IP数据报应当由该节点高层进行处理，上至UDP，在传输层抽取出DHCP请求报文，此时DHCP服务器获取到了DHCP请求报文
  5. DHCP选取一个IP，生成包含这个IP和DNS服务器的IP、默认网关路由器的IP、子网掩码的DHCP ACK 报文，再依次放入到UDP数据报 IP数据报 以太网帧当中，目标MAC即为用户主机的MAC
  6. 由于交换机为自学习的，之前从用户主机发送的已经被记录，因此按照原本的记录即可转发到用户主机
  7. 用户主机收到后，依次抽取 IP数据报 UDP数据报 DHCP ACK 报文，DHCP客户端记录下IP和子网掩码和DNS服务器的IP，在转发表中安装默认网关

2.DNS和ARP

此时，便可以进行互联网的访问了，输入www.google.com的URL,此时，只知道域名，故目前应当做的便是通过DNS解析出对应的IP地址和使用ARP解析出MAC地址，最终将请求定位到谷歌的服务器之上

  1. 用户主机生成一个DNS查询报文，将www.google.com 放入到问题段当中，封装成UDP报文，该UDP报文封装到目标IP为DNS服务器的IP数据包当中
  2. 将该IP数据报放入到以太网帧当中，在上一步知道了网关路由器的IP，为获取MAC地址，则需要使用到ARP
  3. 生成一个目标IP为网关路由器的ARP查询报文，将该报文放入到目标MAC为FF:FF:FF:FF:FF:FF(广播)的以太网帧当中，向交换机发送该以太网帧，交换机将其广播给所有的设备
  4. 网关路由器连向用户主机内网的接口收到了该以太网帧，在ARP表当中查询到了对应的MAC地址，准备一个ARP响应报文，放入到以太网帧当中，目标MAC即为用户主机网卡的MAC地址。通过交换机转发至用户主机处
  5. 用户主机从以太网帧当中抽取出网关路由器的MAC地址，用户的主机此时便定位到了网关路由器，准备将跨越子网向DNS服务器发送请求
  6. 用户主机将包含DNS查询的以太网帧寻址到网关路由器的MAC地址，通过交换机发送至网关路由器处

3.域内路由选择到DNS服务器

  1. 网关路由器的连接用户子网的接口抽取处IP数据报，检查目标IP，通过路由表确定出接口，将其通过链路层转发到Comcast网络的路由器上(转发表通过BGP协议+OSPF/RIP协议事先确定)
  2. 此时以太网帧到达Comcast网络的网关服务器的接口，该接口检查目标服务器，通过转发表确认出接口向DNS服务器发送数据报(转发表通过OSPF/RIP协议事先确定)
  3. 最终，包含DNS查询的数据报到达了DNS服务器，依次从中解析接IP数据报，UDP数据报，DNF查询报文，通过缓存/迭代查询/递归查询的方式查询出该域名对应的IP,生成一个DNS应答报文。放入UDP数据报当中，再放入目标IP为用户主机的IP数据报当中，通过Comcast反向转发到学校路由器，并经过以太网交换机抵达用户主机
  4. 用户主机依次抽取IP数据报、UDP数据报、获取到DNS响应报文，获取www.google.com的IP地址，准备向谷歌的服务器建立连接

4.Web客户-服务器交互

  1. 用户主机已经获取到了服务器的IP，但由于HTTP协议是基于TCP协议的，因此首先需要进行TCP三次握手建立稳定的连接：用户主机生成一个TCP SYN报文段(HTTP 目的端口为80 HTTPS目标端口为443) 放入目标IP为谷歌IP的IP数据报当中，再放入目标MAC为网关路由器的以太网帧中，发送到网关路由器
  2. 学校、Comcast、谷歌网络中的路由器通过转发表，将该数据报进行转发
  3. 最终该数据报抵达谷歌的服务器，从中抽取出 TCP SYN报文，并分解到与80/443端口相联系的欢迎套接字，并生成一个连接套接字。产生一个SYNACK报文段，放入目标IP为用户主机IP的IP数据报，再放入以太网帧当中
  4. 该数据报依次通过谷歌网络、Comcast网络、学校网络。抵达用户主机的网卡，从中解析出SYNACK报文，数据报分解到18中生成的TCP套接字，客户端进入连接状态，再按照同样的步骤发送一个报文段，完成TCP三次握手
  5. 浏览器产生一个 HTTP GET请求报文，写入到套接字当中，GET报文成为TCP报文的Payload，TCP报文放置到IP数据报当中，向www.google.com按照上述步骤进行发送，
  6. 谷歌HTTP服务器从TCP套接字中读取HTTP GET报文，将请求的web放入到HTTP响应体当中，把报文发送到套接字
  7. 包含HTTP响应的报文经过一系列转发，到达用户主机，浏览器从套接字中读取HTTP响应，抽取出HTML，进行渲染，展示。
