Git

分布式版本控制

基本原理：

  1. 每个库都有完整的提交历史，可以直接在本地进行提交代码
  2. 每次提交记录为完整的文件快照，而不是增量记录
  3. 通过Push等操作来完成和远端代码的同步

基本使用方式

目录

  • HEAD中存储的为当前指向的分支、

    $ cat .git/HEAD
ref: refs/heads/master

  • config为配置信息

  • object为文件信息

  • refs分支信息

工作区&暂存区

工作区为平时修改代码的位置，通过git add 命令提交到暂存区，git commit提交到git目录

配置

config

分为 system global local三个级别，高级别的会覆盖低级别的配置

  • git config --global user.name "fischer" global级别配置用户名
  • git config --global url.git@github.com:insteadOf https//github.com/instead of配置，使用SSH替代HTTP

remote

添加远程仓库后：

$ git remote add origin git@github.com:Fischer0522/myPostBar.git


$ git remote -v
origin  git@github.com:Fischer0522/myPostBar.git (fetch)
origin  git@github.com:Fischer0522/myPostBar.git (push)

分为fetch和push的源

添加远程仓库的源后，配置文件也发生相应的变化：

$ cat .git/config
[core]
        repositoryformatversion = 0
        filemode = false
        bare = false
        logallrefupdates = true
        symlinks = false
        ignorecase = true
[remote "origin"]
        url = git@github.com:Fischer0522/myPostBar.git
        fetch = +refs/heads/*:refs/remotes/origin/*

HTTP&SSH

通过HTTP配置在连接时提示输入密码进行身份验证，但可以采用配置进行免密认证：

git config --global credential.helper --timeout='3600' 存储在内存中，过期时间为3600s,也可配置存储在硬盘当中

2021年之后不再支持使用HTTP进行配置，输入完密码后会提示

SSH免密配置使用公私钥的机制，默认为rsa，但由于安全问题，不推荐dsa，rsa，优先推荐使用ed25519，

Objects

  • Blob：存储文件的内容
  • Tree：存储的为文件目录信息
  • commit：存储提交信息，一个commit对应一个提交代码
  • Tag： 存储附注标签信息

串联

  1. 先通过Commit寻找到Tree信息，每个Commit对应一个Tree ID

    $ git cat-file -p 767242a1145e1fba00b5538c1149713b5cebd9a3
tree 386eb799bbfd0826f510b6587c5afaae2a962279
author fischer <1809327837@qq.com> 1660114623 +0800
committer fischer <1809327837@qq.com> 1660114623 +0800

add readme.md

  2. 通过Tree存储的信息，获取对应的目录树信息

    $ git cat-file -p 386eb799bbfd0826f510b6587c5afaae2a962279
100644 blob 2be7c65ae93b54b988416f280298b0b8b5f20385    readme.md

  3. 从tree中获取blob的ID，通过Blob ID获取文件的内容(blob id即为执行git add 时新增文件的id)

Refs

refs 内容为对应的commit ID，把ref当作指针，指向对应的commit来表示当前Ref(即对应的分支)对应的版本

`-- refs
    |-- heads
    |   |-- master
    |   `-- test
    `-- tags

$ cat .git/refs/heads/master
767242a1145e1fba00b5538c1149713b5cebd9a3

Branch

git checkout -b name创建一个新分支，在开发时使用。可以通过不断添加commit进行迭代

git tag v创建一个tag 用于发布时表示版本号,表示一个稳定的版本，指向的commit一般不会变更

Annotation tag

附注标签，用于给tag添加额外信息，如新增了什么feature等

修改历史版本

  • commit –amend 修改最后一次commit的信息，修改之后commit ID会改变，执行后进入vim，对信息进行修改
  • rebase：git rebase -i HEAD~3实现对最近三个commit的修改
    • 合并commit
    • 修改具体的commit message
    • 删除某个commit
  • filter –branch ：删除commit中的某一个文件等操作

使用amend操作后，新的修改在新的commit ID上，旧的commit则称为了没有ref指向的悬空的commit：使用git fsck --lost-found进行查看

$ git fsck --lost-found
Checking object directories: 100% (256/256), done.
dangling blob e69de29bb2d1d6434b8b29ae775ad8c2e48c5391
dangling commit 3501f03ecdf290572f2b8641b068f0dcf3eb4298

GitGC

  • git gc 可以删除一些不需要的object，并对object进行打包压缩减少仓库体积
  • Reflog 用于记录操作日志，手动将日志设置为过期
  • git gc prune=now 指定修剪多久之前的对象，默认为两周

远端同步

  • clone：拉取完整的仓库到本地
  • fetch：拉取远端的分支到本地，不会进行merge
  • pull：等于 fetch+merge
  • push

工作流

  • GitFlow
  • Github Flow
  • GitLab Flow

Github Flow

只有一个主干分支，基于Pull Request往主干分支提交代码

本地进行push之后，可以直接使用返回的url提交一个PR，也可在github上手动创建一个PR来合并

代码合并

  • Fast-Forward：不会产生一个merge节点保持线性历史
  • 三方合并：产生一个新的Merge节点
