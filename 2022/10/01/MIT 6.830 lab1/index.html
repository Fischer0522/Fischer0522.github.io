判断slot是否被使用
只需找到header中对应那一个bit即可，由于Header采用字节存储，因此涉及到bit 和byte之间的转化

public boolean isSlotUsed(int i) {
        // some code goes here
        // 先获取到对应的字节，在从字节中找到对应的bit
        int index = i / 8;
        // 偏移量，用于确认在第几位bit
        int offset = i % 8;
        // 与偏移n位的1位与得到第n位bit
        return (this.header[index] & (1 << offset)) != 0;
    }

统计未使用过的slot的数量
通过不断和与自身-1进行位与，可以得到该字节中有几个1

public int getNumEmptySlots() {
        // some code goes here
        int numUsedSlot = 0;
        for(byte b :header) {
            // 统计出该bit中有多少个1，即多少个slot被使用
            while (b != 0) {
                numUsedSlot ++;

                b &= (b-1);
            }
        }
        return getNumTuples() - numUsedSlot;
    }

迭代器
由于page中存在部分slot并未使用，因此迭代器应当跳过这些未使用的slot，无法直接Arrays.asList(arr).iterator(),需要自己实现一个迭代器，根据前面写的判断该slot是否被使用的方法来跳过未使用的slot

public Iterator<Tuple> iterator() {

        // some code goes here

        return new Iterator<Tuple>() {
            private int nextSlot = 0;
            @Override
            public boolean hasNext() {
                // getNumTuples() - getNumEmptySlots()为当前已经使用的slot的数量
                return nextSlot < getNumTuples() - getNumEmptySlots();
            }

            @Override
            public Tuple next() {

                if (!hasNext()) {
                    throw new NoSuchElementException();
                }
                    while (!isSlotUsed(nextSlot)) {
                        // 获取被使用的slot
                        nextSlot++;
                    }
                    return tuples[nextSlot++];
            }
        };
    }

迭代器：
迭代器需要实现读取所有的该文件中的所有的tuple，但是在一个HeapFile中又有多个HeapPage，因此需要做的便是遍历每个page，再在page中调用HeapPage中封装好的迭代器，去迭代遍历整个page中的所有tuple

@Override
    public DbFileIterator iterator(TransactionId tid) {
        // some code goes here
        return new HeapFileIterator(this,tid);
    }

    private static final class HeapFileIterator implements DbFileIterator {
        private  final HeapFile heapFile;
        private static Iterator<Tuple> it;
        private final TransactionId transactionId;
        // 当前正在迭代的Page
        private int whichPage;

        public HeapFileIterator(HeapFile heapFile,TransactionId transactionId) {
            this.transactionId = transactionId;
            this.heapFile = heapFile;
        }
        @Override
        public void open() throws DbException, TransactionAbortedException {
            // 初始化，从第0页开始迭代
            whichPage = 0;
            it = getPageTuples(whichPage);
        }

        private Iterator<Tuple> getPageTuples(int pageNumber) throws TransactionAbortedException{
            // 获取用于迭代该Page的一个迭代器
            if (pageNumber >=0 && pageNumber < heapFile.numPages()) {
                HeapPageId heapPageId = new HeapPageId(heapFile.getId(),pageNumber);
                try {
                    HeapPage page = (HeapPage) Database.getBufferPool().getPage(transactionId, heapPageId, Permissions.READ_ONLY);
                    return page.iterator();
                } catch (DbException e) {
                    e.printStackTrace();
                    try {
                        throw new DbException(String.format("heapfile %d does not contain page %d!", pageNumber,heapFile.getId()));
                    } catch (DbException ex) {
                        ex.printStackTrace();
                    }
                }
            }
            return null;

        }

        @Override
        public boolean hasNext() throws DbException, TransactionAbortedException {
            if (it == null) {
                return false;
            }
            if (!it.hasNext()) {
                // 当前页已经迭代完毕，则尝试获取下一页进行迭代
                if (whichPage <heapFile.numPages() -1) {
                    whichPage++;
                    it = getPageTuples(whichPage);
                    // 获取下一页的迭代器，判断下一页中是否有tuple可以继续迭代
                    return it.hasNext();
                } else {
                    // 没有下一页就结束迭代
                    return false;
                }
            } else {
                // 本页未迭代完毕
                return true;
            }

        }

        @Override
        public Tuple next() throws DbException, TransactionAbortedException, NoSuchElementException {
            if (it == null || !it.hasNext()) {
                throw new NoSuchElementException();
            }
            // 令page的迭代器去迭代tuple
            return it.next();
        }

        @Override
        public void rewind() throws DbException, TransactionAbortedException {
            close();
            open();

        }

        @Override
        public void close() {
            it = null;

        }
    }

在测试时发现，MIT提供的dat生成代码存在一定的bug
起初认为是自己的代码存在一定的问题，直到后续查看了二进制文件


创建了一个3X4的txt文件，生成的二进制文件中的header却为7，转换为二进制则为0000 0111,代表只有三个tuple,并且最后一个tuple在二进制文件中也只找到了 0A(10) 0B(11) 但是并没有找到对应的0C
因此最终根据header就只能找到三条数据，查询结果也只有三条数据

原因：

if (c == '\n') {
            if (first) {
                continue;
            }
            recordcount++;
            first = true;
        }

HeapFileEncoder类中有一个readCount来统计具体的行数，而这个readCount的增加是根据换行符来进行判断的，而最后一行本来就无换行符，因此最后一行在统计的时候行数上丢失
根据readCount统计的行数来进行header的写入，因此header的最后一个也自然而然的被遗漏，而读取时是根据header来进行读取的，自然而然就会少读出来一行

for (i=0; i<nheaderbits; i++) {
                if (i < recordcount)
                    headerbyte |= (1 << (i % 8));
                
                if (((i+1) % 8) == 0) {
                    headerStream.writeByte(headerbyte);
                    headerbyte = 0;
                }
            }

对于写入时会漏掉最后一个字符的问题，字符的写入是根据换行符或者分隔符进行判断的，最后一行的前几个会因为分隔符被顺利写入，但是最后一个就无法避免的被遗漏了

if (c == fieldSeparator || c == '\n' || c == '\r') {
           String s = new String(buf, 0, curpos);
           if (typeAr[fieldNo] == Type.INT_TYPE) {
               try {
                   pageStream.writeInt(Integer.parseInt(s.trim()));
               } catch (NumberFormatException e) {
                   System.out.println ("BAD LINE : " + s);
               }
           }

解决方案：为了保证header和最后一个字符均能够被正确写入，因此可以选择在最后额外添加一行，多出的换行符能够保证二者均能够正确写入
