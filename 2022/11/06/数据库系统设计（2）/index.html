查询优化

  • Rules：
    • 重写查询来移除一些无效的东西，如：select * from A where 1 = 1可以重写为select * from A
    • 可能需要检查catalog
  • Cost-Based Search
    • 使用cost model来进行评估执行一个计划的开销
    • 评估多个然后选出开销最低的那个计划

通常先进行rule-based rewriting 然后进行代价评估
架构总览/执行流程

  1. SQL Rewriter
  2. Parser：对SQL进行解析，生成一个抽象语法树
  3. Binder 如查询catalog确认是否有这一张表等操作，最终生成一个逻辑计划
  4. Tree Rewriter
  5. Optimizer：只进行计划之间的估计和比较，不会给出实际的代价 得到最终的物理计划，交给执行器来执行


逻辑计划和物理计划

  • 逻辑计划：从高层次说明这个sql要干什么，但是不说怎么实现，与关系代数或者sql的形式基本一致
  • 物理计划：使用访问路径定义了具体的执行策略 ：用这一个索引，通过hash join进行join等
  • 并非一一对应的：比如逻辑上执行了join和order by，但是如果join底层的实现算法为sort merged join,join的结果已经是排序好的，即不需要再通过order by进行排序

一致性

数据库尝试对现实世界进行建模并且遵循完整的约束，所谓一致性是数据库处理前后结果应与其所抽象的客观世界中真实状况保持一致。这种一致性是一种需要管理员去定义的规则。管理员如何指定规则，数据库就严格按照这种规则去处理数据。
比如通过DB来模拟一个银行系统来完成交易的管理，在进行转账时，如果转帐前A和B账户总计5000元，那么转账后的总计依旧为5000元，此时便称为符合一致性。或者说如果要插入一个年龄为负数的用户，会给予对应的提示让其无法插入
一致性分为数据库一致性和事务一致性

  • 数据库一致性：数据库一致性意味着，如果我在过去做了一次更改，只要没人覆盖掉我的更改，那么未来任何时刻都应当能读取到我这次的更改
  • 事务一致性：我们无法通过DBMS来保证事务的一致性，需要应用程序给予对应的逻辑，比如我给定一个规则是A账户-100，B账户+100，那么DBMS即按照这个规则进行操作，并且保证逻辑上不出现错误

隔离性

多个用户提交多个事务，每个事务都各自运行，互相不存在干扰。
但是DB通过交叉执行各个事务来达到并发的效果，因此我们需要通过一种方式来协调各个事务，保证各个事务在时序上交叉执行，但是在逻辑上又独立运行互不干扰
并发控制协议
并发控制协议是DBMS如何决定在多事务时的正确交叉执行
存在两种协议：

  • 悲观锁：悲观锁：对于同一个数据的并发操作，悲观锁认为自己在使用数据的时候一定有别的线程来修改数据，因此在获取数据的时候会先加锁，确保数据不会被别的线程修改。Java中，synchronized关键字和Lock的实现类都是悲观锁。而在MySQL当中，排他锁即为悲观锁
  • 乐观锁：乐观锁认为自己在使用数据时不会有别的线程修改数据，假设并发冲突很罕见，所以不会添加锁，只是在更新数据的时候去判断之前有没有别的线程更新了这个数据（具体方法可以使用版本号机制和CAS算法）。如果这个数据没有被更新，当前线程将自己修改的数据成功写入。如果数据已经被其他线程更新，则根据不同的实现方式执行不同的操作：重试或者报异常。

悲观锁为实际的锁，而乐观锁为通过协调来保证不会出现冲突，如果出现冲突再尝试解决

在之前的课程中也提到过锁与锁存器的区别，主要可以通过一下的表格来概述二者的区别

锁的概念更加属于数据库的抽象逻辑层面的，意在保护数据库的逻辑结构 ，如tuple或者 表等

锁存器意在保护数据库的物理结构 如真实的数据结构等，如使用锁存器来解决一个B+树的并发问题，所讨论的就是使用锁存器来对那个树的节点来进行加锁保护。

本节课主要讨论的是锁，即数据库的逻辑层面

​

锁的种类，简单来说有以下两种，后续再进行扩展

  • 共享锁（读锁）shared locks for reads S-LOCK
  • 排他锁（写锁） Exclusive locks for writes X-LOCK

数据库通过Lock Manager来进行对锁的控制，维护一个 internal lock-table来记录目前锁的申请和获取 等待情况，从而控制并发：

  • 事务来申请一个锁
  • Lock Manager判断后授予一个锁或者令该事务阻塞等待
  • 事务最后释放锁

通过区分了锁的种类之后，由于S-LOCK 与S-LOCK之间并不冲突，因此提升了一定的并发度，加上锁的种类之后，一次并发流程如下图

两阶段锁协议

顾名思义分为两个阶段：Growing和Shrinking

  • 在Growing阶段中，每个事务按照自己的需要来从lock manager那里去获取锁，Lock Manager也根据锁的冲突原则来赋予或拒绝锁
  • 在Shrinking阶段中，事务只被允许去释放之前所获取的锁，无法再去申请新的锁

​

遵守两阶段锁协议之后，在一次事务的执行过程中，不会存在多个获取-释放的动作，获取和释放的过程被严格分于两个阶段当中，在遵循两阶段锁协议后，上一个例子的执行流程如下：解决了不可重复读的问题

级联中止（Cascading Aborts）

使用两阶段锁协议在协调冲突可序列化时很有效，但是又引入了新的问题，即级联终止

在下图中，T2锁读取的是T1写之后的数据，但是如果T1进行了回滚，放弃掉整个事务，则磁盘和内存中的数据应当还是原本没被T1修改过的值，但是T2读取到的是T1修改过的值，因此T1的终止导致T2发生了脏读，因此T2也应被终止，形成了级联终止T1终止导致T2读取了不存在于存储中的数据，因此T2也被回滚

​

除此之外，两阶段锁协议还会导致死锁的问题，后续再进行讨论

强限制两阶段锁

在原本的两阶段锁协议的基础上再加以限制：

  • Growing阶段与原本的不变
  • Shrinking阶段等到事务提交时才允许释放锁

​

因此只有提交后的事务的数据才对其他事务可见,带来了以下的好处：

  • 不存在级联中止问题
  • 可以通过存储之前的原始值来进行事务的回滚

两阶段锁协议

由于两阶段锁协议，T1会在释放A的X-LOCK之前就先获取B的X-LOCK，T2只有先等待T1释放了X-LOCK(A)之后才能去申请S-LOCK(B)，因此就可以保证对B的更新在ECHO之前

​

强限制两阶段锁协议

使用两阶段锁协议之后，T1在事务提交时才释放掉X-LOCK(A)，因此T2在T1提交之后才进行ReadA的操作，总体上已经比较接近于串行化，相比于普通的两阶段锁协议并行度降低

​

死锁

两阶段锁协议可能会导致死锁

都在尝试获取对方的锁

对于死锁问题，主要采用破除和预防两种手段

​

在等待图中如果出现循环，则证明发生了循环等待，出现死锁

需要终止一个进程来破除循环

当DBMS检测到了死锁，选择出一个牺牲者是来进行回顾来破除循环等待问题

牺牲者进程可能重启也可能直接放弃掉（通常是直接放弃）取决如何调用

由于使用一个后台线程去建立Waits-For Graph​去探寻是否有死锁也存在CPU开销，如果我想尽快发现死锁，那么可能频率是一微秒一次，这就会带来较大的CPU开销，而如果将频率放宽到一秒一次，那么遇到死锁时最长需要等待一秒，但是后台线程的CPU开销就大大降低，在这二者之间需要有一个权衡取舍，究竟是要尽快检测出死锁还是降低CPU的开销用于别的事

牺牲者的选取通常由一下几个因素：

  • 年龄，即事务创建的时间戳
  • 任务，比如如果一个事务需要执行大量的查询，那么最好把他当作牺牲者（最好不要把他当作牺牲者）具体是否选取取决于不同的数据库
  • 持有的锁的数量
  • 回滚时关联的其他的事务的多少（级联终止）强限制两阶段锁并不是必须的，有的系统采用普通的两阶段锁协议，锁定然后级联中止的处理方式

真正的选择牺牲者时并非考虑以上的某一单一因素，而是综合情况，甚至需要建立一个预测模型

死锁预防

对于死锁的处理，可以通过锁的合理分配，来达到提前避免，首先假设时间较久的时间戳的事务拥有较高的优先级，则通常有以下两种策略：

  • Wait-Die（old waits for young）

    • 如果申请锁的比现在持有锁的拥有更高的优先级，那么申请锁的继续等待，直到持有锁的释放锁
    • 其他情况申请的事务放弃（ aborts）

  • Wound-Wait(yong wait for old)

    • 如果申请锁的比现在持有锁的拥有更高的优先级，那么持有锁的放弃（aborts），然后释放锁
    • 其他情况申请锁的等待

在下图中 T1先开启事务，则拥有较早的时间戳，因此有较高的优先级，但是此时T2已经持有了锁，最终的结果即为：

  • Wait-Die：T1 继续等待
  • Wound-Wait：T2 放弃

在下图中 T1先开启事务，则拥有较早的时间戳，因此有较高的优先级，并且此时T1已经持有了锁，最终的结果即为：

  • Wait-Die：T2 放弃
  • Wound-Wait：T2 等待

​

当一个事务进行重启时，为了防止饥饿，被重启的事务保持原有的优先级，否则会一直保持较低的优先级导致一直重启

锁级别

之前讨论的锁都是锁一个tuple，那么如果一个事务需要更新一百万条tuple，那么则需要加100w个锁，显然不合理，因此需要引入锁级别，上述情况下直接锁住整张表，则相当于该表所有的tuple均加上了锁

​

意向锁

意向锁是一种不与行级锁冲突表级锁，因此可以不去检查底层的节点（tuple）是否存在冲突就去对更高一级别的节点去加锁（Table）

意向锁分为两种：

  • 意向共享锁（intention shared lock, IS）：事务有意向对表中的某些行加共享锁（S锁）
-- 事务要获取某些行的 S 锁，必须先获得表的 IS 锁。
SELECT column FROM table ... LOCK IN SHARE MODE;
  • 意向排他锁（intention exclusive lock, IX）：事务有意向对表中的某些行加排他锁（X锁）
-- 事务要获取某些行的 X 锁，必须先获得表的 IX 锁。
SELECT column FROM table ... FOR UPDATE;

即：意向锁是有数据引擎自己维护的，用户无法手动操作意向锁，在为数据行加共享 / 排他锁之前，InooDB 会先获取该数据行所在在数据表的对应意向锁。

  • 共享意向排他锁（Shared Intention Exclusive Lock）：= S-LOCK + IX-LOCK，S-LOCK这一属性表明要去读取表的所有的记录，其他的事务就不会并发的更新任意的记录，IX-LOCK让其他事务知道我们要更新目标表的一部分记录，其他事务就不会并发的读取所有的记录

​

  • 如果想要在一个节点上获取到S或者IS，那么事务必须在父节点（table）上至少持有IS
  • 如果向获取X，IX SIX，那么必须在父节点上至少持有IX

时间戳排序并发控制

​

乐观与悲观的定义是针对于对并发过程中是否存在冲突的看法，悲观锁假定每次访问都存在冲突，因此每次无论读写都需要获取锁来确保正确处理冲突，而乐观锁是假定不存在冲突，将主数据库中需要读取或修改的数据读出一份本地副本，在本地副本上进行操作，最后在提交到主数据库当中，在提交前去检验是否存在冲突，如果存在则再去处理。

对于乐观锁来说，在数据库的逻辑层面上不需要进行任何的显示加锁操作，但是对于底层的数据结构而言，依旧需要锁存器的支持，保证不会进行随意的覆盖。两个事务即便在逻辑上不存在任何的交互，最终在数据结构上也可能存在冲突。

对于每个事务，分配一个时间戳，来表示这个事务的执行时间，时间戳的具体签发时机根据不同的策略而定，时间戳本身有以下特点：

  • Timestamp单调递增
  • 唯一值

每一个事务分配一个独一的时间戳，有以下的实现策略：

  • System Clock
  • Logical Counter
  • Hybrid

基本时间戳顺序协议

首先通过时间戳来对事务进行标记，同时对于一个tuple 需要记录其 W-TS和R-TS，即最后一次对其进行读写操作的事务的时间戳，用于在一个事务要对一个tuple进行读写操作时判断是否有其他事务会和它产生冲突

在基本时间戳顺序协议中，时间戳记录的是该事务开始时的时间，即通过BEGIN关键字声明开启事务时旧分配一个时间戳，但是DBMS本身还是并发的，因此可能会出现后续的事务在较早的事务之前执行，此时就可能存在问题，可能要考虑将较早的事务给撤销，并重新分配时间戳重新执行

当一个事务访问到一个tuple时，发现TS < W-TS ，则证明未来的一次写入覆盖了目前正在读的数值，即出现了脏读问题，读取到的是旧值，需要终止

对于读写，有以下的规则：

  • 读：

    • 如果TS($T_i$) < W-TS(X),则违反了时间戳顺序，读到了旧的值，该事务需要重启并分配新的时间戳

    • 否则：

      • 允许事务T去读取tuple X
      • 更新R-TS（X）为$max(R-TS(X),TS(T_i))$​
      • 生成一个本地副本用于该事务后续的重复读（如果再去读最新的数据则会产生两次读取不一致的不可重复读的问题）

  • 写：

    • 如果$TS(T_i)<R-TS(X) or TS(T_i) < W-TS(X)$ 放弃$T_i$并重启$T_i$

      • $TS(T_i)<R-TS(X)$会导致后续读取操作读取不到此次的写入
      • $TS(T_i) < W-TS(X)$会导致此次的写入无意义

    • 否则：

      • 允许事务$T_i$去写入，并且更新W-TS(X)
      • 生成一个本地副本用于可重复读​

例子：

  1. T1先执行，生成一个时间戳1
  2. T1进行读操作，更新tupleB的R-TS(B)为1
  3. T2事务开始，生成一个时间戳2
  4. T2进行读操作和写操作，将R-TS(B)和W-TS(B)均修改为2，之前的R-TS(B)为1 < 2，因此不存在冲突
  5. T1 读，R-TS(A) =1
  6. T2 读写 R-TS(A) = 2 W-TS(A) = 2

不存在任何冲突问题

​

例2

  1. T1开启事务，分配时间戳1
  2. T1 写，R-TS(A) = 1
  3. T2 开启事务，分配时间戳2
  4. T2 写 W-TS(A) = 2
  5. T1 写A， T = 1 < W-TS(A) 发生冲突 T1事务被放弃

Thomas Write Rule

  • $TS(T_i) <R-TS(X)$:放弃并重启$T_i$（与原本一致）
  • $TS(T_i) <W-TS(X)$:忽略此次写入，并允许事务继续执行
  • 其他情况则未违反协议，允许$T_i$写入，并更新$W-TS(X)$

虽然本质上违反了协议，但是当前此次写后续会被覆盖掉，因此可以直接进行忽略

按照此原则，上图中的T1的W（A）就会被忽略掉，事务T1继续直接执行，只维护一个本地的副本，用于后续的读取操作

如果存在一个长事务，后续新来的短在产生冲突时会一直放弃，最终导致饥饿

可恢复性

一个事务T，假设他所依赖的数据为D，那么如果在T进行commit时，所有同样依赖于D并且位于T之前的事务都已经commit了，则称为可恢复的

A位于B之前，AB都依赖于data D，B commit时A应当已经commit了

如下，T1位于T2之前，T2commit时T1还未commit，此时如果T1放弃了，则T2读到了T1写了的但是放弃了的值，发生了脏读

​

乐观并发控制

DBMS给每个事务均分配一个私有的工作空间：

  • 进行读操作时将读的结果copy到私有的工作空间当中
  • 修改操作则针对于工作空间来进行修改

当一个事务准备提交时，再将工作空间和全局Database进行比较，查看是否存在冲突，这个过程称为Validation​

如果不存在冲突，则将工作空间的结果应用到全局Database当中

阶段

  • Read Phase：当事务进行读写操作时，将所有访问到的数据（tuple）读取到私有的工作空间后，再工作空间上进行操作
  • Validation Phase：在事务commit前验证是否和其他事务存在冲突
  • Write Phase：如果不存在冲突，则将私有工作空间的做的更改应用到全局数据库上，否则放弃并重启事务

在还未进行validate时，副本的时间戳设置为∞，如果进行了写操作，那么此时私有空间中W-TS即为∞

在验证节点才会获取一个时间戳，因此将验证的时间操作视为事务真正的开始时间，A先于B进行验证，则认为A先于B开启事务

例：

  1. T1 开启事务，但不分配时间戳
  2. T1 R(A)，将A copy到自己的私有空间当中
  3. T2开启事务
  4. T2 R(A)，将A copy到自己的私有空间当中
  5. T2 准备提交，进行验证，获取一个时间戳TS = 1 T1的时间戳为正无穷，不存在冲突直接提交
  6. T1 W(A)
  7. T1 验证并提交

对于所有的活动的事务，维护一个全局变量来记录各个事务的状态，让各自的状态都对其他事务彼此可见，用于在validation阶段判断是否存在冲突

幻读

T1进行了两次select max​，但是在两次select​之间进行了一次插入操作，最终导致了不可重复读问题，即称为幻读

在使用两阶段锁协议时会存在幻读，原因为两阶段锁协议只能对当前存在的元素去加锁，因此无法防止别的事务去新插入的别的值

可以通过谓词锁定的方式解决，即锁定所有status = 'lit'​的tuple，但是由于代价开销过高，没有任何一个系统会去使用

除此之外还可以采用索引锁，即在索引上进行加锁，由于索引的有序性，各个值有序排列，通过间隙锁的方式在值与值之间的间隙上加锁，就可以防止新的值的插入，MySQL即采用了间隙锁的解决方案
