MIT 6.830 Lab4

Exercise1 & Exercise2

为实现页级锁，需要定义一个PageLock​​和LockManager​​来进行辅助管理

buffer pool policy策略上，选择了NO-STEAL + FORCE

简单来说就是对于一个为提交的事务，不可将其的操作进行刷盘，而在一个事务commit时，需要将其所有的更改刷盘进行持久化，为最简单的实现方式。最低限度的保证了并发安全，但是并无法实现WAL，WAL采用的buffer pool policy使用的为STEAL + NO-FORCE，具体和buffer pool policy相关的看：数据库系统设计（2）[^1]

锁的管理策略为 Strict two-phase locking​，即只有事务进行commit时再释放所有相关的锁，来避免级联中止的问题。因此需要一个相应的TransactionComplete​方法来释放该事务设计的所有相关的锁

PageLock

针对一个page进行上锁，因此需要记录，锁的类型，还有占有锁的事务tid

提供一个用于修改锁类型的方法用于后续的锁类型升级

package simpledb.Lock;

import simpledb.storage.PageId;
import simpledb.transaction.TransactionId;

public class PageLock {
    public static final int SHARED = 0;
    public static final int EXCLUSIVE = 1;

    private TransactionId transactionId;
    private int type;

    public PageLock(TransactionId transactionId, int type) {
        this.transactionId = transactionId;
        this.type = type;
    }

    public TransactionId getTransactionId() {
        return transactionId;
    }

    public void setTransactionId(TransactionId transactionId) {
        this.transactionId = transactionId;
    }

    public int getType() {
        return type;
    }

    public void setType(int type) {
        this.type = type;
    }
}

LockManager

用于管理锁

首先需要一个concurrentHashMap来管理一个page上有多少个锁，因此该concurrentHashMap的key为PageId​,value为pageLocks​，pageLocks用于记录该page上一共有多少个锁，

对于LockManager本身，需要实现获取锁，查看该page上是否有锁，释放一个page上的所有锁，并且提供一个事务结束的方法，释放该事务对应的所有page上的锁

pageLocks

ConcurrentHashMap<TransactionId, PageLock> pageLock;

用于记录一个page上存在什么锁，自身的数据结构选择为concurrentHashMap类型，来保证线程安全，key为事务id，便于事务commit时可以将该事务所有的锁全部释放，value为一个pageLock对象，表明该page上的锁的实体

由于锁的类型问题，由于一个page上面只能有一个排他锁，因此如果pageLocks的size > 1的话，那么当前该page上面的锁一定全为共享锁，如果当前只有一把锁，那么再去获取并判断该锁的类型

isHoldLock

需保证线程安全，使用synchronized​关键字

判断一个事务是否在该page上加了锁，因此传入一个tid和pageId，

先通过pageId来获取到pageLocks，如果pageLocks为null则证明并不存在任何事务在该page上加了锁，因此可以直接返回false，如果能够通过pageId获取到pageLocks，再通过tid来尝试是否能够获取到锁

public synchronized boolean isHoldLock(TransactionId transactionId, PageId pageId) {
        ConcurrentHashMap<TransactionId, PageLock> transactionIdPageLockConcurrentHashMap = lockMap.get(pageId);
        //如果该page上无锁则可以直接返回
        if (transactionIdPageLockConcurrentHashMap == null) {
            return false;
        }
        // 尝试根据id来获取锁
        PageLock pageLock = transactionIdPageLockConcurrentHashMap.get(transactionId);
        if (pageLock == null) {
            return false;
        }
        return true;

    }

releaseLock

需保证线程安全，使用synchronized​关键字

释放该page上的锁，如果释放成功则返回true，否则返回false，首先尝试获取该page上的锁，如果pageLocks==null​则证明该page上并不存在锁，直接返回false，

如果能够获取成功，则将pageLocks中的锁直接remove，返回true

如果remove后pageLocks.size()==0​则证明该page上面已经不存在锁，将其从lockMap中移除即可

其他情况则视为释放失败，返回false

public synchronized boolean releaseLock(TransactionId transactionId, PageId pageId) {
        // 有锁则逐级释放
        if (isHoldLock(transactionId, pageId)) {
            ConcurrentHashMap<TransactionId, PageLock> transactionIdPageLockConcurrentHashMap = lockMap.get(pageId);

            transactionIdPageLockConcurrentHashMap.remove(transactionId);
            if (transactionIdPageLockConcurrentHashMap.size() == 0) {
                lockMap.remove(pageId);
            }

            return true;
        }
        return false;
    }

acquireLock

尝试在该page上获取一个锁，如果无法获取到一个锁，则直接返回一个false，自旋等待的过程在buffer pool中进行

需保证线程安全，使用synchronized​关键字

尝试获取一个锁，因此需要传入进行上锁的pageId，锁的类型，进行加锁的tid

最简单的情况为当前的page上面没有锁，即通过pageId从lockMap中获取的结果为null，直接先创建一个pageLock，在创建一个pageLocks添加到lockMap当中，返回true，

如果获取到的pageLocks不为null，则证明当前的page上面存在锁，因此需要判断锁的种类来判断是否能获取到锁，如果能则直接获取，获取不到则自选阻塞等待。

先判断当前事务是否在该page上面加过锁，如果没有：

  • 如果pageLocks.size()>1​则证明当前的page上面的全为共享锁，

    • 此时如果新加的锁也为共享锁，则可直接添加上去
    • 如果当前的锁为排他锁，则需要进行自选等待，等待设计的共享锁全部释放

  • 如果pageLocks.size()==1​则该锁可能是共享锁，也可能是排他锁，需要遍历value进行判断：

    • 如果是共享锁，则可以直接添加
    • 如果是排他锁，直接拒绝

如果在该page上已经加过锁了，则需要考虑进行锁升级：

  • 首先获取之前page上的锁的类型，如果是共享锁，则可以考虑进行锁升级

    • 如果当前page上只有一个锁，则是当前事务之前加上去的共享锁，进行锁升级，将其的type设置为排他锁
    • 如果当前page上面不止一个锁，则证明还有其他的事务在该page上面持有共享锁，无法进行锁升级，直接返回false

  • 如果该page上面的锁为排他锁，直接进行拒绝

‍

除了上述的其他情况，则全为意外情况，证明锁获取失败，returnfalse

public synchronized boolean acquiredLock(TransactionId transactionId, PageId pageId, int type) {
        // 先判断是否存在锁，如果不存在则可以直接申请持有一个锁
        ConcurrentHashMap<TransactionId, PageLock> locks = lockMap.get(pageId);
        if (locks == null) {
            // 全局一个锁都无，可以直接申请持有
            PageLock pageLock = new PageLock(transactionId, type);
            ConcurrentHashMap<TransactionId, PageLock> lockConcurrentHashMap = new ConcurrentHashMap<>();
            lockConcurrentHashMap.put(transactionId, pageLock);
            lockMap.put(pageId, lockConcurrentHashMap);
            return true;
        }
        // 判断当前事务之前是否在该page上持有锁
        PageLock pageLock = locks.get(transactionId);
        if (pageLock == null) {
            // 当前page上不存在该事务的锁，直接申请新的锁
            if (locks.size() > 1) {
                // 如果 size() > 1则证明存在多个锁，且为多个共享锁，如果当前事务要申请共享锁，直接进行添加，排他锁则拒绝
                if (type == PageLock.SHARED) {
                    PageLock pageLock1 = new PageLock(transactionId, type);
                    locks.put(transactionId, pageLock1);
                    lockMap.put(pageId, locks);
                    return true;
                } else if (type == PageLock.EXCLUSIVE) {
                    return false;
                }

            } else if (locks.size() == 1) {
                // 当前page上只有一个锁，可能为共享锁也可以为排他锁
                PageLock curLock = null;
                for (PageLock lock : locks.values()) {
                    curLock = lock;
                }
                // 获取锁用于判断类型
                if (curLock.getType() == PageLock.SHARED) {
                    // 如果是共享锁并且该事务也为共享锁则可以申请持有
                    if (type == PageLock.SHARED) {
                        PageLock newPageLock = new PageLock(transactionId, type);
                        locks.put(transactionId, newPageLock);
                        lockMap.put(pageId, locks);
                        return true;
                    } else if (type == PageLock.EXCLUSIVE) {
                        // 当前事务为排他锁，拒绝
                        return false;
                    }

                } else if (curLock.getType() == PageLock.EXCLUSIVE) {
                    // page上的锁为排他锁，直接拒绝
                    return false;
                }

            }

        } else if (pageLock != null) {
            // 之前存在过锁，需要考虑锁的升级
            if (pageLock.getType() == PageLock.SHARED) {
                if (type == PageLock.SHARED) {
                    // 新锁为共享锁，直接返回
                    return true;
                } else if (type == PageLock.EXCLUSIVE) {
                    // 当前事务为排他锁，如果该page上没有其他事务持有的锁，则可以进行申请
                    if (locks.size() == 1) {
                        pageLock.setType(PageLock.EXCLUSIVE);
                        locks.put(transactionId, pageLock);
                        lockMap.put(pageId,locks);
                        return true;
                    } else if (locks.size() > 1) {
                        // 其他事务持有了共享锁，存在冲突，不能进行锁的升级
                        return false;
                    }
                }

            }
            // 之前为写锁，无论如何都无需进行升级，新事务为写锁则直接返回，新事务为共享锁则拒绝
            return pageLock.getType() == PageLock.EXCLUSIVE;
        }
        return false;

    }

completeTranslation

事务提交时，释放该事务持有的所有的锁，遍历page进行释放操作

public synchronized void completeTranslation(TransactionId tid) {
        // 遍历所有的页，如果对应事务持有锁就会释放
        for (PageId pageId : lockMap.keySet()) {
            releaseLock(tid, pageId);
        }
    }

HeapFile

用于在进行insertTuple​时，HeapFile的insertTuple​方法会调用bufferpool当中的getPage​来获取一个page，因此同样会进行加锁操作，但是如果获取到了一个满的page，此时会虽然无法对其进行写操作，但是依旧会加锁，此时便需要将该满page上的锁进行释放，防止影响之后的操作，虽然释放锁的操作违反了严格两阶段锁协议，但是由于并未对其进行读写操作，因此释放锁时并不会造成其他影响。对HeapFile进行修改：

Exercise3

实现NO-STEAL策略，在事务未提交时，该事务所涉及的所有的脏页均不能落盘，因此对之前实现的evictPage​方法进行修改：

  1. 从lruList的尾部去除最长时间未被访问的节点，如果不是脏页则直接淘汰，如果为脏页则跳过，并且由于访问了一次，因此将其置于头部，如果不为脏页则进行淘汰
  2. 最终遍历buffer pool当中所有的page，如果全为脏页，则抛出异常

Exercise4

事务提交时执行transactionComplete​,分为两种，一种为一定会提交，另外一种为可能提交也可能终止，根据传入的参数进行决定

根据FORCE​原则，在提交时需要将所有的脏页均进行刷盘，而如果进行终止，则需要从数据库中读取原本的数据将被该事务修改的数据进行恢复

无论是提交还是终止，均需要释放所有该事务持有的锁

因此，transactionComplete​需要完成的工作为：

  • 处理page：

    • 如果事务提交，则将该事务涉及到的所有的page进行刷盘
    • 如果事务中止，则从硬盘中读取page，将内存中脏页恢复到事务开始前的状态

  • 释放该事务涉及到的所有锁

  public void transactionComplete(TransactionId tid, boolean commit) {

restorePages​用于从硬盘当中重新读取page

public synchronized void restorePages(TransactionId tid) {
    for (LinkedNode linkedNode : pageStore.values()) {
        Page page = linkedNode.page;
        PageId pageId = linkedNode.pageId;
        if (tid.equals(page.isDirty())) {
            int tableId = pageId.getTableId();
            DbFile databaseFile = Database.getCatalog().getDatabaseFile(tableId);
            Page pageFromDisk = databaseFile.readPage(pageId);

            linkedNode.page = pageFromDisk;
            pageStore.put(pageId,linkedNode);
            moveToHead(linkedNode);
        }
    }
}

Summary

lab4可以说是真正进入了数据库的深水区，实现了一个简单的并发管理，lockManager的实现还是有一定的难度，尤其是其中的acquiredLock​方法，需要考虑的情况还是有点复杂的。整体难度对于前三个lab还是有较大的差距。不得不说的是DBMS自身还是有相当的复杂性的，看445时感觉对MVCC的理解也较为透彻，但是真正动手实现一个单版本的两阶段严格锁协议也挺费劲的。

后续再考虑能否将其改为MVCC模式，毕竟两阶段严格锁协议的并发度还是太低了，在一读一写的情况下直接就变为串行化了，读写不能同时进行，可以说是没有并发了。

对于锁的粒度，看看是否能将page级别的锁升级为tuple级别，对于一个tuple进行加锁，进一步提高并发度，优先尝试使用tuple级别的锁，在实现上应该比MVCC能简单一点
