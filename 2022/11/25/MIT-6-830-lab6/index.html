MIT 6.830 Lab6

在simpledb当中，并没有严格区分 日志的种类，将redo log 和undo log放在一起，通过一份日志来完成redo undo操作。

日志记录主要涉及到这五种类型：

static final int ABORT_RECORD = 1;
static final int COMMIT_RECORD = 2;
static final int UPDATE_RECORD = 3;
static final int BEGIN_RECORD = 4;
static final int CHECKPOINT_RECORD = 5;

除UPDATE_RECORD​之外，其他几个均只起标志作用，不会涉及到具体的操作，只有UPDATE​类型才是真正记录对数据库的修改操作

对于ABORT​COMMIT​BEGIN​，中间的内容为空的，而CHECKPOINT​类型会记录在设置检查点的时间内的活跃事务

RollBack

在日志当中，通过setBeforeImage​来保存一次刷盘前的page数据，在进行回滚时读取setBeforeImage​设置的page，并将其进行应用来进行恢复

回滚对一个事务所涉及到的所有的操作进行undo，因此只需要关心UPDATE_RECORD​类型的日志记录

首先通过tidToFirstLogRecord​找到该事务的开始位置，然后进行读取beforeImage​，由于需要撤销所有的操作，因此对于一个page，只需要将最早的beforePage进行应用即可，后续的beforePage都无需去管。因此便可以通过一个Set集合，判断是否对一个page进行undo，如果undo过了就直接不管

Set<PageId> set = new HashSet<>();
Page beforeImage = readPageData(raf);
Page afterImage = readPageData(raf);
PageId pageId = beforeImage.getId();
if (txid == tid.getId() && !set.contains(pageId)) {
    set.add(pageId);
    Database.getBufferPool().discardPage(pageId);
    Database.getCatalog().getDatabaseFile(pageId.getTableId()).writePage(beforeImage);
}

Recovery

在simple-db当中的checkpoint policy为fuzzy checkpoint，因此在进行记录checkpoint时，不会中止其他事务的执行和新事务的开启来强制刷盘，而是通过记录活跃事务的方式，记录在checkpoint过程当中未commit仍在执行的事务。

因此并不能在读取到checkpoint之后，直接跳转到checkpoint的位置，对后面所有的事务进行redo和 undo，会导致遗漏活跃数组在checkpoint之前的操作。

最笨的方法为从日志的头部开始遍历，即忽略掉checkpoint，使用一个set来保存已经提交的事务，对于已经提交了的事务，对其进行redo操作，即写afterImage,而对于未提交的事务，对其执行undo操作，恢复到事务执行前的状态，即写beforeImage

有一个设想时在找到了checkpoint之后，直接进行跳转，读取到活跃事务进行保存，先对checkpoint之后的进行操作，之后再对遗漏的活跃事务补上去，如果活跃事务后续提交了，则对其写afterImage（因为最后一次写入和commit之间可能会被checkpoint打断，因此不能直接对其忽略）而如果一个活跃事务没有commit，则对其进行undo操作，但目前存在的问题是，在db crash之后，内存中维护的tidToFirstLogRecord不存在，因此直接使用checkpoint中记录的位置即可，使用一个hashMap来记录事务以及对应的初始位置。

                        }
                    }
                }


            }
        }

使用checkpoint总过程如下：

  1. analyse：读取checkpoint，分析出活跃事务
  2. 统计出beforePage和afterPage
  3. 如果事务提交，则写入afterPage，如果事务中止，则写入beforePage
  4. 针对活跃事务再去进行redo undo，提交则redo写入afterPage，否则则undo写入beforePage

public void recover() throws IOException {
        synchronized (Database.getBufferPool()) {
            synchronized (this) {
                recoveryUndecided = false;
                // Done
              
                /* Phase 1. Analysis */
                raf.seek(0);
                currentOffset = raf.readLong();
                if (currentOffset >= 0) {
                	raf.seek(currentOffset);
                }
              
                /* Phase 2. Redo */
                Map<Long,Set<PageId>> pageIds = new HashMap<Long,Set<PageId>>();
                Map<PageId,Page> pages = new HashMap<PageId,Page>();
              
                while (raf.getFilePointer()+4 <= raf.length()) {
                	LogRecord record = LogRecord.readNext(raf);
              
                	if (null == record) {
                		break;
                	} else if (record instanceof CheckPointRecord) {
                		for (Long key : ((CheckPointRecord)record).keySet()) {
                			tidToFirstLogRecord.put(key, ((CheckPointRecord)record).getOffset(key));
                			if (!pageIds.containsKey(key)) {
                				pageIds.put(key, new HashSet<PageId>());
                			}
                		}
                	} else if (record instanceof BeginRecord) {
                		tidToFirstLogRecord.put(((BeginRecord)record).tid, (record).offset);
                		pageIds.put(((BeginRecord)record).tid, new HashSet<PageId>());
                	} else if (record instanceof CommitRecord) {
                		for (PageId pageId : pageIds.get(((CommitRecord)record).tid)) {
                			Page page = pages.get(pageId);
                		
                			Database.getCatalog().getDatabaseFile(pageId.
                					getTableId()).writePage(page);
                			page.setBeforeImage();
                		}
                		tidToFirstLogRecord.remove(record.tid);
                		pageIds.remove(record.tid);
                	} else if (record instanceof UpdateRecord) {
                		Page page = ((UpdateRecord)record).afterImage;
                			
                		pages.put(page.getId(), page);
                		pageIds.get(((UpdateRecord)record).tid).add(page.getId());
                	} else if (record instanceof AbortRecord) {
                		currentOffset = raf.getFilePointer();
                		rollback(TransactionId.of(record.tid));
                		pageIds.remove(record.tid);
                		raf.seek(currentOffset);
                	}
                }
              
                /* Phase 3. Undo */
                currentOffset = raf.getFilePointer();
                for (long tid : pageIds.keySet()) {
                	raf.seek(currentOffset);
                	raf.writeInt(ABORT_RECORD);
                    raf.writeLong(tid);
                    raf.writeLong(currentOffset);
                    force();
                    currentOffset = raf.getFilePointer();
                    rollback(TransactionId.of(tid));
                }
            }
         }
    }

因此目前怀疑是不是bufferpool中的flushpages出了什么问题

page.setBeforeImage();

终于找到了罪魁祸首

看一看setBeforeImage的描述，setBeforImage​用于在一个事务commit后，将当前的page复制到beforePage当中，用于之后进行rollback时来读取

   /*
     * a transaction that wrote this page just committed it.
     * copy current content to the before image.
     */
@Override
    public void setBeforeImage() {
        synchronized(oldDataLock)
        {
            oldData = getPageData().clone();
        }
    }

再看看其调用的getPageData​方法，首先创建一个ByteArrrayOutPutStream，用于接受最后的结果，写入header，在根据slot的使用情况，未使用的写入0，使用的则写入真实的数据，非空的slot，遍历写入各个feild，最后转换成字节数组返回，代表这整个page中的数据。最终由oldData进行接受，作为该Page的上一个版本的数据

在HeapPage的构造方法当中，最后调用setBeforeImage，对oldData进行赋值

for (Map.Entry<PageId, Page> entry : pageStore.entrySet()) {

            Page page = entry.getValue();
            page.setBeforeImage();
            if (tid.equals(page.isDirty())) {
                flushPage(page.getId());
		// 错误位置 ：page.setBeforeImage();
            }
        }

在flushPages方法中调用setBeforeImage​，而flushPages​最终是由transactionComplete​进行调用。

再对流程进行分析：

@Test public void TestCommitAbortCommitCrash()
        throws IOException, DbException, TransactionAbortedException {

    doInsert(hf1, 1, 2);

void doInsert(HeapFile hf, int t1, int t2)
        throws DbException, TransactionAbortedException, IOException {
        Transaction t = new Transaction();
        t.start();
        if(t1 != -1)
            insertRow(hf, t, t1);
        Database.getBufferPool().flushAllPages();
        if(t2 != -1)
            insertRow(hf, t, t2);
        t.commit();
    }

首先通过doInsert方法插入两条数据，分别为1和2

在插入1之前，在insert方法中会通过HeapPage的构造方法来获取一个HeapPage来用于进行插入，正如上面所说的，在构造方法中保存了oldData，但为全新创建的page，此时的oldData中涉及到的数据为0

插入1后，此时为脏页，因此刷盘，但调用到的为flushAllPages​，flushAllPages​直接调用flushPage​，因此不会去调用setBeforeImage​,oldData也不会被更新

插入2后，此时也为脏页，进行事务的提交，调用的为flushPages​，来将该事务涉及到的所有page刷盘，此时会调用setBeforeImage​，因此此时的oldData包含了 1 和 2

再通过doInsert方法插入一条数据，值为5，此时进行插入，是直接从buffer pool中进行page的获取，因此不会走构造方法，也不会去setBeforePage​，因此oldData仍为 1 2，

在5插入之后，flushAllPages​进行落盘，flushAllPages​最终调用到flushPage​方法，HeapPage中的真实数据有 1 2 5三条数据，但是oldData中却只有1 2,因此在flushPage​进行写入日志时：

Database.getLogFile().logWrite(dirtier,page.getBeforeImage(), page);
Database.getLogFile().force();

由于oldData中只有 1 2，因此通过getBeforeImage()​所获取到的也只有1 2两条，最终日志中也就只写入了两条数据

之后doInsert方法执行结束事务进行提交，最终通过transactionComplete​然后调用flushPages，如果flushPages中调用了setBeforeImage​方法，那么5就也会被设置到oldData当中，此时一切正常。

但是如果按照错误的写法，即只有脏页才去调用serBeforeImage​，由于在5插入之后已经进行了一次刷盘，因此在最终事务提交时，便不会进行setBeforeImage​，5便被遗漏，oldData当中就只有 1 2，因此后续rollback​再去读取也就只能读取到1 2,这一切也都能解释的通了

再说说为什么调换顺序了就不存在问题，如果调换顺序，即调用doInsert(-1,5)​第一个传入的-1，由于-1并不会被写入，因此buffer pool当中就不存在脏页，因此在进行flushAllPages时，就不会在flushPage中写入日志。然后在事务提交时，5进行了插入，因此当前页为脏页，5就能正常的通过setBeforeImage​被记录到oldData当中。这样就没有bug

如果在在flushPages​中直接不调用setBeforeImage​，那么oldData的结果就一直为0，rollback​之后就直接清空所有数据。

其实文档中已经给出了相应的提示，不能在flushPage​当中直接调用setBeforeImage​，而我那种写法与直接在flushPage​当中进行调用无异，只能说写lab时还是有点粗心，没去仔细阅读文档

(Note: We can’t just call setBeforeImage()​​ in **flushPage()​, since flushPage()​ might be called even if a transaction isn’t committing. Our test case actually does that! If you implemented transactionComplete()​ by calling flushPages()​ instead, you may need to pass an additional **argument to flushPages()​​ to tell it whether the flush is being done for a committing transaction or not. However, we strongly suggest ** in this case you simply rewrite transactionComplete()​ to use flushPage()​.)
